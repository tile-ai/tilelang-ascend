# TileLang算子开发指南：从GPU迁移到NPU硬件优化

## 引言

在异构计算领域，能够快速将算子部署到不同硬件平台是提升开发效率的关键。TileLang通过统一的编程模型屏蔽了底层硬件差异，使得开发者可以用类似CUDA的思维编写算子，并轻松迁移到昇腾NPU等平台。本文将以矩阵乘法为例，首先展示从GPU到NPU的极简迁移过程，随后深入剖析NPU的硬件特性，并针对这些特性给出具体的性能优化策略。通过本文，您将掌握在TileLang中开发高效NPU算子的核心方法。

## 第一部分：从GPU到NPU的平滑迁移

### 1. GPU版本：标准的TileLang矩阵乘法

下面的代码是TileLang官方Quick Start中给出的矩阵乘法示例，主要面向GPU/CPU平台编写：

```python
@tilelang.jit(out_idx=[-1])
def matmul(M, N, K, block_M, block_N, block_K, dtype=T.float16, accum_dtype=T.float32):
    @T.prim_func
    def gemm(
        A: T.Tensor((M, K), dtype),
        B: T.Tensor((K, N), dtype),
        C: T.Tensor((M, N), dtype),
    ):
        # 定义网格：将输出矩阵C划分为 (block_M x block_N) 的块
        with T.Kernel(T.ceildiv(N, block_N), T.ceildiv(M, block_M), threads=128) as (bx, by):
            A_shared = T.alloc_shared((block_M, block_K), dtype)
            B_shared = T.alloc_shared((block_K, block_N), dtype)
            C_local = T.alloc_fragment((block_M, block_N), accum_dtype)

            T.clear(C_local)
            for k in T.Pipelined(T.ceildiv(K, block_K), num_stages=2):
                T.copy(A[by * block_M, k * block_K], A_shared)
                T.copy(B[k * block_K, bx * block_N], B_shared)
                T.gemm(A_shared, B_shared, C_local)

            T.copy(C_local, C[by * block_M, bx * block_N])

    return gemm
```

该代码体现了两个核心设计模式：
- **分核策略**：通过 `T.Kernel` 将输出矩阵划分为多个子块，每个内核（Kernel）负责一个子块的计算。内核数量由 `ceil(M/block_M)*ceil(N/block_N)` 决定。
- **Tiling策略**：在单个核内，沿K维将数据进一步切分，利用 `T.Pipelined` 实现计算与数据加载的流水线重叠。

### 2. NPU版本：仅需三处微调的迁移代码

将上述kernel迁移到NPU只需修改三个细节：

```python
@tilelang.jit(out_idx=[-1], target="npuir")  # 修改1：指定目标平台
def matmul(M, N, K, block_M, block_N, block_K, dtype="float16", accum_dtype="float32"):
    @T.prim_func
    def gemm(
        A: T.Tensor((M, K), dtype),
        B: T.Tensor((K, N), dtype),
        C: T.Tensor((M, N), dtype),
    ):
        # NPU使用一维索引，需手动映射到二维
        with T.Kernel(T.ceildiv(N, block_N) * T.ceildiv(M, block_M), is_npu=True) as (cid, _):
            # 修改2：从一维cid恢复二维by、bx
            by = cid // T.ceildiv(N, block_N)
            bx = cid % T.ceildiv(N, block_N)

            A_shared = T.alloc_shared((block_M, block_K), dtype)
            B_shared = T.alloc_shared((block_K, block_N), dtype)
            C_local = T.alloc_fragment((block_M, block_N), accum_dtype)

            for k in T.Pipelined(T.ceildiv(K, block_K), num_stages=2):
                T.copy(A[by * block_M, k * block_K], A_shared)
                T.copy(B[k * block_K, bx * block_N], B_shared)
                # 修改3：用initC参数替代显式清零
                T.gemm(A_shared, B_shared, C_local, initC=(k == 0))

            T.copy(C_local, C[by * block_M, bx * block_N])

    return gemm
```

**迁移改动总结：**
- 指定 `target="npuir"` 生成NPU代码。
- 分块网格变为一维，通过计算 `cid` 恢复二维坐标。
- 利用 `T.gemm` 的 `initC` 参数在第一次迭代初始化累加器，代替 `T.clear`。

除此之外，内存分配、流水线循环、数据拷贝等核心逻辑完全一致。这充分说明TileLang使得跨平台迁移变得异常简单。

### 3. 运行Kernel与分块参数调整

完成kernel编写后，可以像普通函数一样调用，并自由调整分块参数：

```python
import os
import torch
import tilelang

os.environ['TILELANG_ASCEND_MODE'] = 'Developer'

# 实例化kernel时可灵活指定分块大小
kernel = matmul(1024, 1024, 1024, 128, 128, 32)

a = torch.randn(1024, 1024).npu().half()
b = torch.randn(1024, 1024).npu().half()
c = torch.zeros(1024, 1024).npu().half()

kernel(a, b, c)

# 验证结果
ref_c = a @ b
torch.testing.assert_close(c, ref_c, rtol=1e-2, atol=1e-2)
print("All check passed.")
```

分块参数 `(block_M, block_N, block_K)` 直接影响分核数量（`ceil(M/block_M)*ceil(N/block_N)`）和Tiling循环次数（`ceil(K/block_K)`），是性能优化的关键。下一部分我们将深入探讨如何针对NPU硬件选择最优的分块参数。

## 第二部分：NPU硬件特性与性能优化策略

在成功迁移算子后，我们需要理解NPU的硬件特点，才能写出高性能的kernel。本节将从片上缓存容量、数据对齐要求、存算并行能力以及物理核数限制四个方面，系统介绍NPU特有的优化方法。

### 1. Tiling的根本原因：有限的片上缓存

昇腾NPU的AI Core包含两级关键缓存：
- **统一缓冲区（UB）**：Vector核私有，大小通常为**192KB**。
- **一级缓存（L1）**：Cube核私有，大小通常为**512KB**（以A2系列为例）。

全局内存（GM）容量虽大，但访问延迟高、带宽有限。因此必须将数据切分成能完全装入片上缓存的小块（即Tiling），通过重复利用小块数据来减少GM访问。Tiling策略的核心是选择合适的分块尺寸，确保每个内核所需的数据量不超过对应缓存的容量。

### 2. 通用优化原则：最大化分块尺寸

无论在GPU还是NPU，一个通用原则是：**在缓存不超限的前提下，尽可能增大分块尺寸**。好处包括：
- 提高数据复用率（如K维分块越大，每个元素参与的乘加次数越多）；
- 减少循环迭代次数，降低开销；
- 增加计算密度，更容易隐藏内存延迟。

例如矩阵乘法中，一个内核需要加载A分块（`block_M×block_K`）和B分块（`block_K×block_N`），两者之和应小于L1容量（Cube核）或UB容量（Vector核）。

### 3. NPU特有的优化考量

#### 3.1 尾轴对齐限制

NPU的加载指令要求数据地址按**32字节**对齐。若张量最后一维（连续存储的维度）大小不是32字节的整数倍，编译器会自动填充，导致：
- 实际占用的片上缓存比预期大（例如`float16`尾轴为15时，会按16加载，多占6.7%）；
- 填充数据浪费带宽和缓存，极端情况（如尾轴为1）性能急剧下降。

因此，分块尺寸应满足对齐要求：
- `float16`：最后一维是16的倍数；
- `float32`：最后一维是8的倍数；
- `int8`：最后一维是32的倍数。

在矩阵乘法中，应确保 `block_K`（A的尾轴）和 `block_N`（B的尾轴）满足上述条件。

#### 3.2 存算并行与流水线深度

NPU支持数据搬运与计算重叠，即存算并行。在TileLang中通过 `T.Pipelined` 循环实现，编译器会自动生成流水线代码。但流水线效率依赖于循环迭代次数：若 `ceil(K/block_K)` 过小（如2~3），流水线无法充分填满，建立和排空开销占比大，性能提升有限，且会额外占用片上缓存（乒乓缓冲）。通常建议该值至少为4~6，以保证流水线稳定运行。

#### 3.3 物理核数限制与分核策略优化

GPU拥有数十至数百个SM，可轻松支持数千个线程块并发。而昇腾NPU的AI Core数量较少（A2系列Cube核约20~24个，Vector核数量翻倍）。虽然运行时允许下发大量逻辑内核（如65535），但超出物理核数的部分会被串行调度，引入额外的核启动开销。因此需要根据数据规模调整分核策略：

- **数据规模中等**：通过增大 `block_M` 和 `block_N` 减少启动的内核总数，使其接近物理核数的整数倍（如20、40、60），避免负载不均（如启动21个内核将导致其中一个物理核执行两倍任务）。
- **数据规模极大**：无法通过调整分块减少下发内核数时，应**固定启动内核数为物理核数**，并在每个物理核内串行处理多个逻辑块的任务，从而摊薄核启动开销。

下面给出实现“核内串行”的示例：

```python
@tilelang.jit(out_idx=[-1], target="npuir")
def matmul(M, N, K, block_M, block_N, block_K, dtype="float16", accum_dtype="float32"):
    num_physical_kernels = 24  # 实际应通过接口获取
    num_logical_kernels = (N // block_N) * (M // block_M)

    @T.prim_func
    def gemm(
        A: T.Tensor((M, K), dtype),
        B: T.Tensor((K, N), dtype),
        C: T.Tensor((M, N), dtype),
    ):
        with T.Kernel(num_physical_kernels, is_npu=True) as (kernel_id, _):
            # 每个物理核负责的逻辑内核数量
            num_local_tasks = T.ceildiv(num_logical_kernels - kernel_id, num_physical_kernels)

            for task_id in T.serial(num_local_tasks):
                cid = task_id * num_physical_kernels + kernel_id
                by = cid // T.ceildiv(N, block_N)
                bx = cid % T.ceildiv(N, block_N)

                # 以下为每个逻辑内核的计算（与原始kernel相同）
                A_shared = T.alloc_shared((block_M, block_K), dtype)
                B_shared = T.alloc_shared((block_K, block_N), dtype)
                C_local = T.alloc_fragment((block_M, block_N), accum_dtype)

                for k in T.Pipelined(T.ceildiv(K, block_K), num_stages=2):
                    T.copy(A[by * block_M, k * block_K], A_shared)
                    T.copy(B[k * block_K, bx * block_N], B_shared)
                    T.gemm(A_shared, B_shared, C_local, initC=(k == 0))

                T.copy(C_local, C[by * block_M, bx * block_N])

    return gemm
```

该模式通过将大量逻辑内核任务均匀分配给固定数量的物理核，避免了核启动风暴，显著提升了大规模问题的性能。

## 总结

本文通过矩阵乘法算子，完整演示了从GPU到NPU的TileLang迁移流程，并深入探讨了NPU硬件特性带来的优化要点。核心优化思路可归纳为：
- **对齐先行**：确保分块尺寸满足32字节对齐，避免编译器填充浪费。
- **流水线深度**：保持足够的Tiling循环次数，发挥存算并行优势。
- **核数适配**：根据物理核数调整分核策略，数据量大时采用核内串行。

掌握这些技巧后，您就可以在TileLang中自如地编写高效NPU算子，充分利用昇腾硬件的计算能力。未来TileLang还将提供更智能的自动调优功能，进一步降低手动优化的复杂度。
