## Fixed Core: Fixed Physical Compute Cores and L2 Resident Cache

In the SparseMLA operator, the AIC (Cube) responsible for matrix operations and the AIV (Vector) responsible for vector processing are independent execution units. They need to frequently interact with a large number of local state tensors (such as `S`, `P` and `O`) during calculation iterations. Under the default mechanism, if the Workspace for Cube-Vector data interaction is dynamically allocated along with the huge logical `block_num`, the memory footprint will expand sharply. Once the allocated space exceeds the capacity of the on-chip cache, these frequently accessed intermediate data will be swapped to the far-end HBM, thereby encountering severe memory bandwidth bottlenecks.

To address this issue, the physical core is explicitly constrained by using the `with T.Kernel(core_num, is_npu=True) as (cid, vid)` syntax:
- **Calculation tasks are statically mapped to physical cores**: Since A2 and A3 machines have 24 physical AI Cores, regardless of how many logical blocks are generated by the input, they are allocated to fixed 24 physical cores through static load balancing. This avoids redundant executions of operations like alloc buffer and annotate_address.
- **Workspace Memory Optimization and L2 Residence**: Physical core binding strictly limits the total capacity of the Workspace used for cross-core data interaction to 24 copies. This fixed and compact memory allocation allows intermediate data to be completely resident in the high-speed on-chip L2 Cache. After the AIC calculation results are written to L2, AIV can read and consume them directly, avoiding the overhead of frequent swaps of intermediate tensors to the HBM.

## Sparse KV: Sparse Memory Access Optimization

In sparse scenarios, KV data is discretely distributed in Global Memory (GM), but subsequent Cube matrix operations require continuous memory input. The following mechanism design and optimization are adopted here:

- **Dual Vector Core Memory Access Instruction Emission**: Based on the 1:2 ratio of Cube cores to Vector cores on A2/A3 chips, the Vector core is used to emit memory access instructions, doubling the instruction issuing efficiency.
- **Discrete KV Gather and Continuous Copy Out**: Uses a unified sparse index to first gather the discrete KV from GM to UB. After gathering into continuous blocks here, they are copied out in one go to the Workspace for subsequent Cube core usage.
- **Asynchronous Copying (Manual Intra-Core Synchronization)**: Manually setup operations for `set_flag` / `wait_flag` make the discrete copy-in executed asynchronously, significantly reducing the overall KV gather time.
- **Copy In and Out Ping-Pong (Double Buffer)**: A double-buffer mechanism is used for the discrete KV gather operation. While the current data is being written to the Workspace through MTE3, the MTE2 read instruction for the next block of data has already been asynchronously initiated, enabling instruction overlap and hiding the latency of MTE2 and MTE3.

## Tiling: Split Strategy

Tiling not only determines the inter-core load balance but also directly affects overall computational efficiency and task scheduling. It mainly involves two key dimensions of split design:

- **head_num Inter-Core Splitting**: The `head_num` dimension is split with fine granularity to generate more parallel executable tasks, fully utilizing the physical AI Cores to enhance overall parallelism and computational throughput.
- **seq_len_kv Intra-Core Splitting**: Increasing the block size of the `seq_len_kv` dimension expands the data scale of each round of Gather KV and Copy Out, which helps to improve memory access throughput. At the same time, enlarging the basic block size of the Cube can further improve its computational efficiency.

## CV pipeline: Overlap Operations of Cube Core and Vector Core

A2 and A3 machines adopt a separated Cube core and Vector core architecture, making CV operation overlapping possible. The `T.Pipelined` primitive introduces dual stages on the `seq_len_kv` loop. The compiler arranges the CV serial tasks within the loop into a CV pipeline execution:
- The **Vector MTE memory access** of the next iteration (like KV gather and copy-out in the V0 stage) overlaps with the **Cube core calculation** of the previous iteration (QK matrix multiplication in the C1 stage); and the **vector calculation** of the next iteration (Online Softmax in the V1 stage) overlaps with the **Cube core calculation** of the previous iteration (PV matrix multiplication in the C2 stage), etc. Using different hardware units to mask each other drastically reduces the operator's execution time.

## Vectorization: Broadcast and AXPY

In the implementation of sparse attention, the Vector core pivots around the dynamic update of Softmax and the progressive accumulation of output results. Early implementations used scalar index acquisition and row-by-row calculation, which introduced additional scaler calculations, provided low vectorization levels, and failed to fully utilize the computing capability of the Vector core.

- **Broadcast Vectorization**: Introduction of intermediate tensors such as `m_i_broadcast`, `m_i_prev_broadcast`, and `sumexp_broadcast` expands row-by-row updated states into shapes aligned with the input tensors via broadcast operations, avoiding row-wise operations and enhancing the execution efficiency of vector instructions.
- **AXPY Instructions**: During the output tensor (`acc_o`) update, the step of "historical result scaling + current result accumulation" with iteration dependencies is refactored into a continuous vector `mul` and `add` pipeline computation via AXPY (a·X + Y) instructions, reducing the time spent on issuing instructions and intermediate memory access.

## Split-K pipelined GEMM v0 implementation: MTE1 and Cube Pipeline Overlap

In standard matrix multiplication execution, data must first be carried over from L1 to L0A/L0B (performed by MTE1) before being offloaded to the Cube compute unit (MMA sequence instruction) for calculation. Under completely synchronous sequential logic, the Cube compute unit sits idle while waiting for subsequent data loading. The `gemm_v0` template adopts the following mechanisms:

- **K-axis Data Blocking and Ping-Pong Double Buffering**: In the inner loop, split the K dimension into fixed-capacity sub-blocks (e.g., `kL0Size = 128`), and allocate double buffers for the A and B matrix sub-blocks in L0 memory. Dynamically alternate writing addresses using `pp = (kL0Idx & 1)`.
- **Fine-Grained Synchronization**: Manual scheduling of synchronization signals ensures that when the Cube compute unit runs the `mma` instruction calculation on the $i$-th chunk of L0 data, the MTE1 unit simultaneously copies the $i+1$-th chunk of data from L1 into the other half of the L0 buffer.

Through the tuned `gemm_v0` template, the MTE1 memory access inside the Cube core is effectively overlapped with the Cube computation, successfully hiding the vast majority of the MTE1 memory access latency of the operator.
